# quarto-book — Overlay Module (Quarto Books)

## Canonical guidance (required)
- Treat the official **Quarto Books** documentation as canonical: https://quarto.org/docs/books/ ([quarto.org](https://quarto.org/?utm_source=openai))
- When providing advice, prefer approaches consistent with Quarto’s documented project model (e.g., `_quarto.yml`, project type `book`, `quarto render`, `quarto preview`) and explicitly note deviations.

## Scope & assumptions (required)
Confirm early (do not infer):
1) output targets (HTML / PDF / ePub / MS Word) and which are required vs optional,
2) computation engine (Knitr/R, Jupyter/Python, none) and whether code execution is allowed in CI,
3) citation workflow (BibTeX/BibLaTeX, CSL, reference manager),
4) publishing target (GitHub Pages / Posit Connect / internal site / other),
5) whether the book is a “living” book (frequent updates) vs a versioned release artifact.

## Project structure expectations (required)
- Assume a Quarto book is a **Quarto project** with a root `_quarto.yml`.
- Use `project: type: book` in `_quarto.yml` and keep the `book:` config in one place. ([quarto.org](https://quarto.org/docs/blog/posts/2026-03-31-typst-books-and-more/?utm_source=openai))
- Prefer a clear chapter structure:
  - `index.qmd` as the entry (unless there’s a documented reason to use a different index),
  - chapters listed explicitly and ordered intentionally,
  - appendices separated explicitly (when applicable).
- Keep project-level assets organized and discoverable (figures, data, styles, filters).

## Configuration discipline (required)
- Treat `_quarto.yml` as the single source of truth for:
  - book metadata (title/author/date),
  - chapter ordering,
  - format options (html/pdf/epub),
  - cross-reference numbering and structural options.
- Prefer project-level variables for repeated values:
  - use `_variables.yml` + `{{< var ... >}}` when appropriate to avoid duplication. ([quarto.org](https://quarto.org/docs/authoring/variables?utm_source=openai))

## Reproducibility & computation (required)
- If the book contains executable code:
  - specify how it is executed (local vs CI),
  - make execution deterministic where possible (seeds, pinned inputs),
  - avoid network-dependent examples unless explicitly allowed and robustly handled.
- If execution is not allowed in CI, document the policy clearly and recommend alternatives (precomputed outputs, cached artifacts, or a separate “build” pipeline).

## Cross-references, citations, and navigation (required)
- Prefer Quarto-native mechanisms for:
  - citations/bibliography,
  - cross-references (figures/tables/sections),
  - TOC and navigation.
- Require a consistent citation style and bibliography configuration across the entire project.

## Output formats & tradeoffs (required)
When asked about format decisions, present options (HTML/PDF/ePub/Word) with tradeoffs:
- HTML: best navigation/search; easiest iteration
- PDF: print-ready; LaTeX/Typst toolchain considerations
- ePub: e-readers; stricter content constraints
- Word: stakeholder review workflows; styling constraints
(Confirm which are required before recommending a toolchain.)

## Publishing & deployment (required)
- Recommend a publishing approach consistent with the target:
  - local preview via `quarto preview`,
  - build via `quarto render`,
  - deploy using the chosen platform’s recommended method.
- For automated publishing, require a clear “definition of done”:
  - what branch/artifact is deployed,
  - versioning strategy (tags/releases),
  - how failures are detected and diagnosed.

## Style, maintainability, and reviewability (required)
- Prefer patterns that make reviewer work easier:
  - consistent naming for chapters and files,
  - small, focused chapters (or subchapters) rather than monoliths,
  - consistent callouts, code chunk conventions, and figure/table captions.
- Avoid undocumented custom hacks (filters, templates) unless justified and documented.

## When uncertain (required)
- Do not speculate. Ask for missing constraints or point to the relevant Quarto documentation section.
- Provide 3–5 options with pros/cons and key tradeoffs (per `chat-manual`).
